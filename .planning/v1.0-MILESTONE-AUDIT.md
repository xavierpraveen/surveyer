# Surveyer v1.0 Milestone Audit (Third Run)

**Audited:** 2026-03-16 (third run — after Phase 8 response+role fixes)
**Status:** ⚠️ gaps_found — 2 new critical blockers discovered by integration checker

## Executive Summary

Phase 8 successfully fixed BUG-03 (response_answers column names) and BUG-04 (analytics.ts + middleware.ts + auth.ts role guards). However the integration checker found that the BUG-04 pattern was **not propagated to Phase 04's Server Actions** — `publication.ts`, `actions.ts`, `settings.ts`, and `tagging.ts` all contain the same raw role comparison bug.

- **BUG-05**: `publication.ts` — leadership/hr_admin/survey_analyst cannot publish survey results
- **BUG-06**: `actions.ts` / `settings.ts` / `tagging.ts` — 11 guard sites block elevated roles from admin operations

These are surgical fixes: the pattern is identical to what Phase 08 already solved — replace `['admin'].includes(role)` with `normalizeRole(user.app_metadata?.role as string | undefined) !== 'admin'`.

## Critical Blockers

### 🔴 BUG-05: publication.ts Role Guard Uses Raw JWT Role

**Severity:** Critical — leadership/analyst/hr_admin cannot publish survey results
**Pattern:** Same as BUG-04 — raw role comparison not propagated from analytics.ts fix

**File:** `src/lib/actions/publication.ts`

| Function | Current guard | Correct guard |
|----------|--------------|---------------|
| `createPublicationSnapshot` | `['admin'].includes(role)` | `normalizeRole(raw) !== 'admin'` |
| `getPublicationSnapshot` | `['admin'].includes(role)` | `normalizeRole(raw) !== 'admin'` |
| `getPublishedCycles` | `['admin'].includes(role)` | `normalizeRole(raw) !== 'admin'` |

**Impact:** PUBLISH-01 through PUBLISH-05 — the entire publication snapshot flow returns "Forbidden" for the roles most likely to use it.
